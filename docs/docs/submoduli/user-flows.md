# Пользовательские потоки (подробно)

Связка **экранов** и **системных** действий. Технические типы маршрутов: `src/navigation/types.ts`.

## 1. Холодный старт приложения

1. Нативный splash → `App.tsx` грузит шрифты (`useAppFonts`).
2. Пока шрифты не готовы — пустой экран; после — `hide` splash.
3. Поднимаются провайдеры: тема, язык, React Query (с персистом части запросов), сессия, еда, база знаний.
4. `KnowledgeProvider` может асинхронно подтянуть JSON (локально и/или по URL).
5. `RootNavigator` читает `useAuth()`: **нет сессии** → стек **Auth**; **есть** → **Main tabs**.
6. При входе в main: `useAppBootstrap` + загрузчик клубов; до готовности пользователь видит состояние загрузки (не «пустые табы» без данных).

## 2. Регистрация

1. Экран регистрации: телефон/логин, пароль — по требованию бэка.
2. Запросы к API регистрации (часто vibe/iCafe); при успехе — переход к verify SMS или сразу к логину.
3. `request-sms` / альтернативный path — env; тело `member_id` / `phone` — env.
4. Ввод кода → при успехе сессия в SecureStore, переход в main tabs.

## 3. Вход (существующий пользователь)

1. Логин + пароль (или схема, которую требует API).
2. Сохранение `AuthContext` + `sessionStorage`.
3. Main tabs, bootstrap данных.

## 4. Сценарий «клубы → зал → бронь»

1. Вкладка **Клубы** — список, фильтр, гео.
2. Выбор клуба — детализация, **схема зала** (если API отдал `structRooms` и т.д.).
3. Переход к **Брони** (таб) с уже выбранным контекстом или выбор клуба внутри брони — зависит от UX текущей версии.
4. На экране брони: дата (календарь), слот по времени, **тариф** (продукты API), **зона**, **ПК** (свободен/занят по live API).
5. Подтверждение → POST с подписью → сообщение об успехе/ошибке.
6. Опционально: **добавить в календарь** (разрешения в `app.config` / Android manifest через Expo), **напоминание** (уведомления).

## 5. «Мои брони» и отмена

1. Список броней с бэка (путь `allBooksPath` в `extra`).
2. Отмена — мутация/запрос на API; обновление кэша React Query; пересчёт уведомлений (`TodayBookingNotificationSync`).

## 6. Пополнение баланса

1. Из профиля — переход в WebView или форму; URL: `EXPO_PUBLIC_TOP_UP_URL` или сборка от `apiBaseUrl`.
2. `topup` / `fetchBonus` — тела и заголовки зависят от `EXPO_PUBLIC_TOPUP_*` (Bearer vs private_key, product_id, …).
3. Ошибка «Member not found» — смотрите env-подсказки в `.env.example` (составной id, минимальное тело).

## 7. Помощь (чат)

1. Пользователь вводит вопрос.
2. Поиск по **структурированной** базе: совпадения по ключам/тексту без вызова внешнего LLM.
3. При отсутствии hit — вежливый отрицательный сценарий (формулировка в i18n).

## 8. Диплинк из уведомления

1. Система открывает `bbplay://…` (см. [modules-map.md](../modules-map.md)).
2. `NavigationContainer` сопоставляет path с табом/стеком.
3. Для **брони** возможен объект `prefill` (клуб, ПК, дата) — ускоряет повторное бронирование.

## 9. Выход

1. Кнопка выхода в профиле/настройках.
2. Очистка сессии, сброс чувствительных query, возврат к **Auth** стеку.
