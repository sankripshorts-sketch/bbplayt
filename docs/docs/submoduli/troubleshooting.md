# Устранение неполадок

Порядок работы: **(1)** воспроизведение, **(2)** логи Metro / Logcat / Xcode, **(3)** сеть и env, **(4)** сравнение с контрактом API. Общий чек-лист: [run.md](run.md) (кэш, Metro, устройства).

## 1. API / 403 / «Api not allowed»

**Симптомы:** HTTP 403, тело/сообщение про запрет API на topup, `fetchBonus`, `PUT` профиля, движения, сессии.

**Почему:** на **прокси** BlackBears для данного **токена** не открыт соответствующий путь. Клиент формирует запросы корректно, если не подменён URL и сессия жива.

**Что сделать:**

1. Проверить, что **логин** тот же контур, что и у тестовой учётки на бэке.
2. Сверить **точный path** в логе с [api-spec.md](../../api-spec.md) и [API_VIBE_LOGIC.md](../../API_VIBE_LOGIC.md).
3. Передать бэкенду: какой path, какой `member_id` / `Authorization` — **открыть путь** для теста.
4. **Не** пытаться «починить» только сменой произвольных `EXPO_PUBLIC_*` без согласования с сервером.

## 2. Бронь не создаётся

**Симптомы:** 4xx, «неверная подпись», пустой успех при ожидании деталей, «ПК занят», «время в прошлом».

**Чек-лист:**

| Шаг | Действие |
|-----|----------|
| Секрет | `EXPO_PUBLIC_BOOKING_SIGN_SECRET` / `VERIFY` совпадает с `SECRET` на прокси (как в реф. Android) |
| Режим ключа | `EXPO_PUBLIC_BOOKING_KEY_MODE` = `android` / `rand_secret` / `legacy_concat` согласован с бэком |
| Сессия | В теле/подписи участвуют **актуальные** `member_id`, `private_key` из ответа логина |
| Имя ПК | `EXPO_PUBLIC_BOOKING_PRESERVE_PC_NAME_CASE=1` если бэк чувствителен к регистру |
| Успех ответа | `EXPO_PUBLIC_BOOKING_LOOSE_SUCCESS` — только если сервер **не** возвращает тело, но код успеха |
| Время | МСК/серверное время: см. `serverBookingClock`, `bookingTimeUtils` |
| Сырой лог | Временно логировать тело **ошибки** (без публикации в открытом issue с секретами) |

Исходник логики: `src/api/bookingKey.ts`, [API_VIBE_LOGIC.md](../../API_VIBE_LOGIC.md).

## 3. Пустые клубы, пустой список броней, «вечная загрузка»

1. **Сеть** и VPN.
2. **`EXPO_PUBLIC_API_BASE_URL`** — без лишнего слеша в конце, если в коде он конкатенируется с путями.
3. **Сессия** — разлогин/логин, проверка что токен не просрочен.
4. **Персист** React Query: часть запросов **не** сохраняется; после офлайн-режима сначала может быть пусто до refetch.
5. В **Dev Menu** React Native: reload; в Metro: `npx expo start -c`.

## 4. Push-уведомления не приходят

- Разрешения в настройках ОС (Android 13+ — разрешения уведомлений).
- **Expo Go** ограничивает ряд сценариев — для полной проверки используйте **development build** или **release**.
- Канал уведомлений (Android) создан в коде/при первом планировании.
- FCM / APNs ключи в EAS, правильный **package** / **bundle id**.

## 5. Metro / сборка

```bash
npx expo start -c
```

При полной «поломке» `node_modules`:

```bash
rm -rf node_modules
npm install
```

На Windows: `rmdir /s /q node_modules` или проводник.

## 6. Схема зала: зоны/ПК не на местах

- Не масштабировать «на глаз» в рантайме: использовать **`EXPO_PUBLIC_HALL_MAP_*`**: глобальные оффсеты, per-club JSON, per-zone, per-PC.
- Сверка с сырыми координатами API (`structRooms`); doc — комментарии в `.env.example`.

## 7. TypeScript / lint

- `npm run lint` — пройти в CI-стиле до коммита.
- Конфликт версий `@types` — выровнять `package.json` / lockfile, переустановить.

## 8. WebView: белый экран, редиректы

- `EXPO_PUBLIC_TOP_UP_URL` укажет **куда** грузить страницу (иногда важен тот же домен, что и сессия куки).
- Для **VK** CORS: `EXPO_PUBLIC_VK_WALL_PROXY_TEMPLATE` на web/некоторых сетях.
