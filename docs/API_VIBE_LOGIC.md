# Логика внешнего API (vibe.blackbearsplay.ru)

Документ описывает **функциональные домены** и **смысловые связи** между методами, а не полный справочник запросов. Детальные контракты, примеры JSON и таблицы полей см. в исходном справочнике (например `API_VIBE (1).md` из материалов конкурса).

**База:** `https://vibe.blackbearsplay.ru` (iCafeCloud, клубы BlackBears Play).

---

## 1. Авторизация и доверие к аккаунту

| Назначение | Эндпоинты | Логика |
|------------|-----------|--------|
| Вход по логину/паролю | `POST /login` | Возвращает профиль участника (`member`) и `private_key` при успехе (`code` часто **3**, не только 0). Это первичный способ узнать `member_id`, клуб (`member_icafe_id`), баланс в ответе. |
| Резервный поиск пользователя | `GET /api/v2/cafe/{cafeId}/members` | Если `/login` недоступен или нужен поиск по клубу — список участников; по нему ищут аккаунт и актуальный баланс (баланс надёжно снимается именно из списка по `member_id`). |
| SMS и верификация | `POST /request-sms`, `POST /verify` | Цепочка «отправить код → подтвердить» для привязки телефона / доверия; бизнес-правило: часть сценариев требует `is_verified` до операций. |
| «Я сам» по токену | `GET /api/v2/cafe/{cafeId}/memberSelf` | Профиль текущего пользователя при `Authorization: Bearer`. |

**Связь с остальным API:** без `member_id` и привязки к клубу (`icafe_id` / `cafeId`) большинство операций (бронь, баланс, история) не имеют контекста.

## 2. Регистрация и участники клуба

| Назначение | Эндпоинты | Логика |
|------------|-----------|--------|
| Новый участник | `POST /api/v2/cafe/{cafeId}/members` | Создание аккаунта в конкретном клубе; ответ «уже существует» — отдельная ветка UI. |
| Список участников | `GET /api/v2/cafe/{cafeId}/members` | Единственный надёжный способ получить **баланс** по `member_id` среди всех полей. |
| Инфо по логину | `GET /member-info-by-account` | Уточнение профиля в разрезе клуба без полного обхода списка. |

## 3. Баланс, бонусы, пополнение

| Назначение | Эндпоинты | Логика |
|------------|-----------|--------|
| Баланс из списка | `GET /api/v2/cafe/{cafeId}/members` | Основной источник чисел по счёту. |
| Баланс «в реальном времени» | `POST /api/v2/cafe/{cafeId}/getRealTimeBalance` | Альтернатива POST для актуального значения. |
| История движений | `GET /api/v2/cafe/{cafeId}/memberBalanceHistory` | Аудит операций по счёту. |
| Бонус перед пополнением | `POST /api/v2/cafe/{cafeId}/members/action/fetchBonus` | Узнать бонусную надбавку до зачисления. |
| Зачисление | `POST /api/v2/cafe/{cafeId}/members/action/topup` | Прямое пополнение (`topup_ids`, `topup_value`, опционально бонус). |
| Платёжка (внешняя) | `POST .../getTopupUrl`, `POST .../getTopupStatus` | Редирект на оплату и проверка статуса транзакции. |

**Типичный флоу пополнения:** `fetchBonus` → `topup` → снова `GET members` (или getRealTimeBalance) для отображения нового баланса.

## 4. Бронирование ПК (ядро домена «бронь»)

Здесь сходятся **тарифы**, **доступность ПК**, **подпись запроса** и **ответ iCafe**.

### 4.1 Подготовка данных

| Шаг | Эндпоинты | Логика |
|-----|-----------|--------|
| Тарифы и пакеты | `GET /all-prices-icafe` | Часовые сетки, пакеты, шаг старта (`step_start_booking`), техперерыв — чтобы UI знал цену и ограничения по времени. |
| Свободные ПК | `GET /available-pcs-for-booking` | Кто свободен на выбранный слот (`dateStart`, `timeStart`, `mins`). |
| Структура зала | `GET /struct-rooms-icafe` | Зоны и типы мест для фильтров и отображения. |
| Точная стоимость слота | `GET /init-booking-session` | Инициализация сессии брони с параметрами `cafeId`, `memberId`, `pcName`, дата/время/длительность. |

**Зоны и цены:** в `/all-prices-icafe` у почасовых строк поле `group_name` задаёт зону (VIP, Game Zone, Bootcamp, …). Пустой `group_name` означает **общий** тариф; в клиенте bbplay он подставляется только если для выбранной зоны нет своей строки (как в iCafe / реф. Android).

### 4.2 Создание брони

| Эндпоинт | Логика |
|----------|--------|
| `POST /booking` | Основное создание брони на один ПК. Тело: `pc_name`, `member_account`, `member_id`, `start_date`, `start_time`, `mins`, `rand_key`, **`key`** (MD5 по правилам ТЗ). Ответ успеха часто с **внешним `code: 3`** и вложенным `iCafe_response` с паролем брони, `member_offer_id`, стоимостью. **Гостевая бронь:** `guest_booking: 1`, `member_id` может быть `null`. |
| `POST /booking-batch` | Несколько ПК одним запросом (например, «себе и другу») — общие `start_date`/`start_time`/`duration_min`, массив `bookings` с разными `pc_name` и участниками. |
| `POST /push-client-status` | Вспомогательное уведомление стороны клуба/ПК о статусе клиента после логики брони. |

**Подпись `key`:** в клиентах BlackBears (реф. Android, bbplay) используется **MD5** от `memberId + randKey + privateKey + secret` — см. [`src/api/bookingKey.ts`](../src/api/bookingKey.ts); нужны секрет (`EXPO_PUBLIC_BOOKING_SIGN_SECRET` или общий с verify) и `private_key` из ответа логина в режиме по умолчанию. В отдельных текстах ТЗ встречается другая конкатенация (с `icafeId`, `pc_name`, датой/временем слота и т.д.) — ориентируйтесь на фактический прокси/бэкенд. Без корректного `key` бронь не примут.

**Ошибки:** занятость слота — в т.ч. `error_600` / сообщения про «occupied»; при **`code: 0`** нужно отличать успех с данными от «успеха-оболочки» с текстом ошибки (см. раздел 7 и клиент `parseVibeBookingPost`).

### 4.3 После брони: списки и отмена

| Эндпоинт | Логика |
|----------|--------|
| `GET /all-books-cafes` | Все брони пользователя по **всем клубам**, ключ верхнего уровня — `icafe_id`. **Цены здесь не приходят**; статус «активна/прошла» на клиенте (например, конец слота + буфер доигрывания). |
| `GET /api/v2/cafe/{cafeId}/bookings` | Список броней в разрезе одного клуба (с параметром вроде `memberAccount`). |
| `DELETE /api/v2/cafe/{cafeId}/bookings` | Отмена по телу JSON: `pc_name`, `member_offer_id` (iCafe Cloud; в bbplay — с тем же базовым URL, что и логин). Устаревший `POST /booking-cancel` на прокси часто недоступен. |

## 5. Клубы и онлайн-инфраструктура

| Назначение | Эндпоинты | Логика |
|------------|-----------|--------|
| Список клубов | `GET /cafes` | Справочник адресов и `icafe_id`. |
| Клуб пользователя | `GET /icafe-id-for-member` | К какому клубу привязан участник. |
| ПК онлайн «сейчас» | `GET /api/v2/cafe/{cafeId}/pcs` или `GET .../pcs/action/member_pcs` | Текущее состояние ПК в момент запроса. В bbplay: `fetchLivePcsForUi` (сначала `member_pcs`, при ошибке шлюза — `pcs`). **Не** источник доступности на выбранный слот. |
| Слот брони | `GET /available-pcs-for-booking` (vibe) | Кто свободен на `dateStart` / `timeStart` / `mins` — только этот метод. |

## 9. Коды ответов и «ловушки»

- **`code: 3`** — нормальный успех для **логина** и часто для **успешной брони** (наряду с вложенным `iCafe_response`).
- **`code: 0`** — и успех, и ошибка в зависимости от тела: если есть **строковое сообщение об ошибке** или `error_600`, это не успех. Для брони клиентский код (`vibeClient.ts`, `parseVibeBookingPost`) разбирает вложенный `iCafe_response`, строковый JSON, поля `booking_password` / `member_offer_id` / `pcs`, а также иные признаки созданной брони (`booking_id`, `offer_id` и т.п., по факту ответа).
- **HTTP 200 с пустым телом** в отдельных методах трактуется как успех (см. общее описание API).

## 10. Прокси (браузер / CORS)

В веб-клиентах запросы могут идти через `proxy.php` с маппингом коротких имён (`endpoint=booking`, `all-prices-icafe`, …) на полные пути vibe — логика API не меняется, меняется только точка входа.
