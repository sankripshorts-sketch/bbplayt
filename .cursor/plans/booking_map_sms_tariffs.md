# План: схема, SMS, тарифы, синхронизация брони (итерация)

## Ранее согласованные блоки

1. **Схема клуба** — измерение ширины канвы (`onLayout`), ужесточение геометрии в `clubLayoutGeometry.ts`, заголовок зоны absolute.
2. **SMS** — расширенный разбор полей + дефолт `next_request_sms_time` при наличии `encoded_data`.
3. **Тарифы** — только с сервера; подписи без дубля «Default»; fallback на строку Default при несовпадении зоны.
4. **Бронь** — `refetch` после успеха для списков броней и доступности ПК.

## 6. Уведомления для компьютерного клуба (идеи)

Ниже — что имеет смысл для гостей и для удержания; часть требует только клиента (локальные push), часть — договорённости с бэкендом или админкой iCafe.

### Локальные (планировщик на устройстве, уже в духе `bookingReminders`)

| Уведомление | Зачем |
|-------------|--------|
| За N минут до начала брони | Не опоздать; напоминание с адресом клуба и местом |
| В момент начала сессии | «Можно занимать место» |
| За M минут до конца оплаченного/забронированного слота | Время продлить или освободить место |
| Напоминание «завтра бронь» вечером | Снизить no-show |

### Серверные / push (нужны события или polling + FCM/APNs)

| Уведомление | Зачем |
|-------------|--------|
| Бронь подтверждена / отменена администратором | Актуальное состояние без ручного обновления |
| Изменение или отмена брони клубом | Шторм, перенос зала |
| Акции и пакеты (скидка на ночь, турнир) | Маркетинг, удержание |
| Низкий баланс / пора пополнить | Если в приложении есть привязка к личному счёту |
| «Место освободилось» в избранном клубе | Если бэкенд умеет waitlist по слоту |
| Турнир / ивент в выбранном клубе | Из раздела новостей — таргет по `icafe_id` |

### Операционные (для доверия к приложению)

- Тихий retry при сбое создания брони + одно уведомление «бронь не создана, нажмите повторить».
- После успешной оплаты/брони — короткое подтверждение с номером брони или паролем из ответа API (если приходит).

### Приоритет внедрения (прагматично)

1. Довести уже запланированные **напоминания до/после брони** (настройки N/M в prefs).
2. **Push от сервера** только для «бронь изменена/отменена», когда появится стабильное событие или webhook.
3. Маркетинговые — отдельная категория с **opt-in** в настройках, чтобы не раздражать.

## Заметки по реализации (не блокируют основной план)

- Локальные: `expo-notifications` + триггеры по времени из данных брони (уже частично есть).
- Серверные: хранение `push_token`, эндпоинт регистрации токена, рассылка по событию из CRM/iCafe (отдельная задача на бэк).
